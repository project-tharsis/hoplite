# WK v1.2 Predicate Mining 增强规格

日期：2026-05-20
状态：待实现
受众：实现代理

## 1. 背景

WK v1.2 规则蒸馏第一版已经完成。

当前结论：

- b-003 prompt blind spots 在 94 条 clean subset 上达标。
- WK v1.2 rule distillation pipeline 已能执行 5 个 phase。
- 当前输出 `wk_rule_candidates.json` 为空数组。
- `rejected_candidates.json` 有 14 条 rejected。
- `wk_candidate_replay.json` 显示 `passes=false`，且 0 candidates 时 before/after 完全一致。

这不是 WK v1.2 失败。

这说明当前 predicate 抽取方式过粗：

- group-level intersection 太保守。
- 很多分歧可能只在子簇内有规律。
- 当前 feature view 不足以把所有分歧解释成高精度规则。

下一步应该增强 predicate mining，而不是实现 WK v1.2。

## 2. 产品目标

建立一个更细粒度的 predicate mining 阶段，使系统能够：

1. 诊断 14 条 rejected candidates 为什么失败。
2. 在每个 rejected group 内搜索更小、更精确的 feature 子簇。
3. 产出新的 WK v1.2 candidate 或明确证明当前 feature 粒度不足。
4. 继续通过 dry-run replay 与 regression manifest 做保守验证。
5. 仍然不修改 `weak_labeler.py`，不写 `data/knowledge.json`。

目标不是降低门槛。

目标是用更好的 predicate 搜索，找到真正满足门槛的确定性规则。

## 3. 非目标

- 不重新运行评估器B。
- 不继续修改 prompt blind spots。
- 不修改 `src/labels/weak_labeler.py`。
- 不写入新的 `weak_labels`。
- 不把 rejected candidate 强行升级。
- 不降低 WK patch proposal 门槛。
- 不引入机器学习模型或 embedding 服务。
- 不依赖外部 API。

## 6. Phase 1：Rejected Candidate 诊断

```bash
python scripts/self_iterate.py diagnose-predicate-mining \
  --kb data/knowledge.json \
  --adjudication data/self_iteration/runs/b-003/adjudication_report.json \
  --rejected data/self_iteration/runs/wk-v1.2/rejected_candidates.json \
  --output data/self_iteration/runs/wk-v1.2-predicate-mining/predicate_mining_diagnostics.json
```

输出结构：

```json
{
  "source": "wk-v1.2 rejected candidates",
  "total_rejected": 14,
  "failure_breakdown": {
    "support_too_low": 0,
    "precision_too_low": 0,
    "false_positive_too_high": 0,
    "empty_predicate": 0
  },
  "by_target": {
    "overall_signal": 0,
    "dimension_signals.execution": 0,
    "dimension_signals.satisfaction": 0,
    "model_signals.1": 0
  },
  "by_direction": {
    "upgrade": 0,
    "downgrade": 0

```

要求：

- 统计每个 rejected candidate 的失败原因。
- 一个 candidate 可以同时计入多个失败原因。
- `empty_predicate` 应该为 0；上一轮已修复 predicate extraction。
- 输出 target/direction 分布。
- 输出 examples 与 counterexamples 的 feature 差异摘要。

验收：

- 诊断文件可解释为什么当前没有 candidate 通过。

## 7. Phase 2：Feature Search Space 构建

新增命令：

```bash
python scripts/self_iterate.py build-predicate-search-space \
  --kb data/knowledge.json \
  --adjudication data/self_iteration/runs/b-003/adjudication_report.json \
  --output data/self_iteration/runs/wk-v1.2-predicate-mining/feature_search_space.json
```

### 7.1 基础 feature

保留现有 deterministic feature view：

```text
result
opponent_quality
venue
competition_stage
score_margin
clean_sheet
dominant_control
dominant_chance_quality
low_defensive_risk
narrow_win
loss_despite_dominance
xg_present
missing_features
```

### 7.2 新增派生 feature

新增以下可解释 feature，全部从现有 `features` 中派生：

```text
big_win                    = result == W and score_margin >= 2
single_goal_win             = result == W and score_margin == 1
failed_to_win               = result != W
strong_xg_edge              = xg_delta >= 1.0
moderate_xg_edge            = xg_delta >= 0.5
shot_volume_edge            = shot_delta >= 8
shot_quality_edge           = shot_on_target_delta >= 3
sterile_possession_risk     = possession_delta >= 8 and xg_delta < 0.5
defensive_leak              = goals_conceded >= 2 or opponent_shots_on_target >= 5
set_piece_edge              = corner_delta >= 4 or set_piece_goals_for > set_piece_goals_against
late_sub_window             = latest_sub_min > 80
early_sub_window            = earliest_sub_min <= 60
sub_impact_positive         = goals_after_arsenal_subs > 0
```

缺失字段语义：

- 缺失字段不能让 boolean 变成 `true`。
- 每个 derived feature 需要记录 required input fields。
- 如果 input 缺失，写入 `missing_features`。

### 7.3 输出结构

```json
{
  "feature_schema_version": "predicate_search_space_v1",
  "rows": [
    {
      "match_id": "1208154",
      "features": {
        "result": "W",
        "big_win": true,
        "strong_xg_edge": true
      },
      "missing_features": ["pressing", "transition"]
    }
  ],
  "feature_definitions": {
    "big_win": {
      "type": "boolean",
      "required_inputs": ["result", "score_margin"]
    }
  }
}
```

验收：

- b-003 clean subset 应有 94 rows。
- 每个 derived feature 有 definition。
- 没有 feature 依赖外部 LLM。

## 8. Phase 3：增强 Predicate Mining

新增命令：

```bash
python scripts/self_iterate.py mine-enhanced-wk-predicates \
  --kb data/knowledge.json \
  --adjudication data/self_iteration/runs/b-003/adjudication_report.json \
  --search-space data/self_iteration/runs/wk-v1.2-predicate-mining/feature_search_space.json \
  --diagnostics data/self_iteration/runs/wk-v1.2-predicate-mining/predicate_mining_diagnostics.json \
  --output data/self_iteration/runs/wk-v1.2-predicate-mining/enhanced_wk_rule_candidates.json
```

### 8.1 搜索对象

只搜索 b-003 clean subset 中的 disagreement rows：

```text
wk_too_harsh
wk_too_generous
dimension_level_disagreement
model_level_disagreement
```

优先级：

1. `wk_too_harsh`
2. `dimension_level_disagreement`
3. `model_level_disagreement`
4. `wk_too_generous`

### 8.2 Predicate 枚举策略

对每个 `(target, direction, target_signal)` group，枚举 1 到 4 个 feature 的组合。

候选 feature 来源：

- categorical exact match:
  - `result`
  - `opponent_quality`
  - `venue`
  - `competition_stage`
- boolean true:
  - 所有基础和新增 derived boolean features

不枚举：

- arbitrary numeric thresholds
- free text
- player names
- opponent names
- missing feature names

数值阈值必须来自 Phase 2 中定义好的 derived feature。

### 8.3 子簇评分

对每个 predicate，计算：

```json
{
  "support": 0,
  "matched_total": 0,
  "precision_vs_b": 0.0,
  "false_positive_count": 0,
  "counterexample_count": 0,
  "cross_run_stability": 0.0,
  "coverage_ratio": 0.0
}
```

定义：

- `support`：predicate 命中的目标 disagreement rows 数。
- `matched_total`：predicate 命中的 clean subset rows 总数。
- `precision_vs_b = support / matched_total`。
- `false_positive_count = matched_total - support`。
- `coverage_ratio = support / group_size`。
- `cross_run_stability`：命中 examples 在 b-001 是否也是 disagreement status。

### 8.4 Candidate 门槛

普通 WK candidate：

- `support >= 3`
- `precision_vs_b >= 0.80`
- `false_positive_count <= 1`
- `cross_run_stability >= 0.60`
- `coverage_ratio >= 0.20`
- predicate size <= 4

进入 implementation spec 的门槛仍然更高：

- `support >= 5`
- `precision_vs_b >= 0.80`
- `false_positive_count <= 1`
- regression replay pass

说明：

本阶段允许输出 `support=3 or 4` 的 exploratory candidate。

但它们只能进入 enhanced candidate artifact，不能直接生成 WK implementation spec。

`wk_too_generous` candidate 仍使用更高门槛：

- `support >= 5`
- `precision_vs_b >= 0.90`
- `false_positive_count == 0`
- `cross_run_stability >= 0.80`

### 8.5 输出结构

```json
{
  "candidate_schema_version": "enhanced_wk_rule_candidate_v1",
  "candidates": [],
  "rejected_candidates": [],
  "search_summary": {
    "groups_scanned": 0,
    "predicates_evaluated": 0,
    "candidates_found": 0,
    "rejected": 0
  }
}
```

验收：

- 每个 candidate 有 deterministic predicate。
- 每个 candidate 有 examples 和 counterexamples。
- 每个 rejected candidate 写明 `rejection_reason`。
- 不允许空 predicate。
- 不允许 predicate 只包含 `result=W` 这种过宽规则进入 candidates。

复用现有 replay 逻辑，新增输入支持 enhanced candidate schema。

命令：

```bash

  --kb data/knowledge.json \
  --adjudication data/self_iteration/runs/b-003/adjudication_report.json \
  --candidates data/self_iteration/runs/wk-v1.2-predicate-mining/enhanced_wk_rule_candidates.json \
  --output data/self_iteration/runs/wk-v1.2-predicate-mining/enhanced_wk_candidate_replay.json
```

要求：

- 如果 enhanced candidates 为空，before 必须等于 after。
- 如果有 exploratory candidate，replay 必须区分：
  - `implementation_eligible`
  - `exploratory_only`
- 只有 `implementation_eligible` candidate 能影响 final pass/fail。

验收：

- `passes` 写入 artifact。
- `candidate_impacts` 列出每条受影响 match。
- 1531572 不得变成 overall 🟢。
- lower/mid_table loss 不得变成 overall 🟢。

## 10. Phase 5：Predicate Mining Summary

新增命令：

```bash
python scripts/self_iterate.py summarize-predicate-mining \
  --diagnostics data/self_iteration/runs/wk-v1.2-predicate-mining/predicate_mining_diagnostics.json \
  --candidates data/self_iteration/runs/wk-v1.2-predicate-mining/enhanced_wk_rule_candidates.json \
  --replay data/self_iteration/runs/wk-v1.2-predicate-mining/enhanced_wk_candidate_replay.json \
  --output data/self_iteration/runs/wk-v1.2-predicate-mining/predicate_mining_summary.json
```

输出必须给出明确分支：

```json
{
  "decision": "proceed_to_wk_v1_2_spec | collect_more_features | no_action",
  "reason": "...",
  "candidate_count": 0,
  "implementation_eligible_count": 0,
  "feature_gaps": []
}
```

决策规则：

- 如果有 implementation eligible candidate 且 replay passes：
  - `decision = proceed_to_wk_v1_2_spec`
- 如果没有 candidate，但 diagnostics 显示 precision 普遍低于 0.5：
  - `decision = collect_more_features`
- 如果有 exploratory candidate 但 support 不够：
  - `decision = no_action`
  - reason 写明等待更多比赛数据

## 11. 如果仍然 0 candidates

如果增强 mining 后仍然 0 candidates，不要继续调门槛。

输出应明确：

```text
当前 feature 粒度不足以把剩余 WK/B 分歧蒸馏成 deterministic WK rule。
下一步应补充 features，而不是降低规则门槛。
```

优先补充 feature：

1. goal timing / score state at goals
2. lead protection quality
3. xG overperformance / underperformance
4. substitution effect by score state
5. opponent shot quality pressure
6. late-game control after leading

这些 feature 必须先进入 `MatchFeatures` 和 backfill/replay pipeline，再重新尝试 WK v1.2。

## 12. 测试要求

新增测试文件：

```text
tests/test_predicate_mining_enhancement.py
```

必测：

1. diagnostics 覆盖所有 rejected candidates。
2. feature search space 只从 existing features 派生。
3. 缺失 input 时 derived boolean 为 false。
4. predicate 枚举不会生成空 predicate。
5. predicate size 不超过 4。
6. `result=W` 单独不能进入 candidates。
7. precision 使用 `support / matched_total`。
8. `wk_too_generous` 使用更高门槛。
9. exploratory candidate 不会触发 implementation spec。
10. replay 空 candidates 时 before == after。
11. replay 写入 `passes`。
12. summary 正确输出三种 decision。

## 14. Done Definition

本阶段完成时，应该能回答：

1. 当前 rejected candidates 为什么没过门槛？
2. 更细粒度 predicate 搜索是否找到了高精度子簇？
3. 如果没有找到，是因为数据量不足，还是 feature 粒度不足？
4. 是否有足够证据进入 WK v1.2 implementation spec？

只有第 4 个问题答案为 yes，才允许进入 WK v1.2 实现。
