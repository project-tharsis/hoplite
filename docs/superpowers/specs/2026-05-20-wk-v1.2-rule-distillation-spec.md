# WK v1.2 规则蒸馏规格

日期：2026-05-20
状态：待实现
受众：实现代理

## 1. 背景

低人工自迭代第一阶段已经完成。

当前关键状态：

- 102 条 feature-backed 比赛已经可以进入 WK/B 对比。
- WK 当前版本为 `weak_label_version=v1.1`。
- prompt-level blind spots 已通过 b-003 验证。
- b-003 有 102 条评估器B输出，其中 94 条通过质量门，8 条被 quarantine。
- `compare-runs` 已支持 clean-subset 对比。

b-003 clean-subset 结果：

```text
common_match_ids: 94
b-001 overall agreement:   64.9%
b-003 overall agreement:   79.8%
b-001 dimension agreement: 27.7%
b-003 dimension agreement: 50.0%
b-001 model agreement:     19.2%
b-003 model agreement:     42.6%
wk_too_harsh:              33 → 13
dimension_disagreement:    35 → 28
```

结论：

> v2/v3 prompt blind spots 在 94 条 clean subset 上达标。

下一步不是继续改 prompt，也不是重新跑评估器B。

下一步是把 b-003 后仍稳定存在的 WK/B 分歧，蒸馏成 deterministic WK v1.2 规则候选。

## 2. 产品目标

建立一个保守的 WK 规则升级流程，使系统能够：

1. 固化 b-003 clean-subset 作为 prompt blind spot 验证证据。
2. 从 b-003 clean subset 的剩余 WK/B 分歧中挖掘 deterministic 规则候选。
3. 对候选规则做 dry-run replay，验证是否提升 WK 与评估器B的一致性。
4. 生成 `wk_v1.2_rule_candidates.json`，供人工审批。
5. 只有候选通过更高门槛后，才进入单独的 WK v1.2 implementation spec。

目标不是让评估器B直接改写 WK。

目标是让 WK 学到可解释、可回放、可测试的确定性规则。

## 3. 非目标

- 不重新运行评估器B。
- 不继续升级 prompt-level blind spot。
- 不在本阶段修改 `src/labels/weak_labeler.py`。
- 不写入新的 `weak_labels` 到 `data/knowledge.json`。
- 不把评估器B结果写成 `human_override`。
- 不迁移 JSON 到数据库。
- 不把单场分歧当成规则。
- 不允许候选规则绕过 replay 和 regression list。

## 4. 核心原则

### 4.1 Prompt 侧和 WK 侧分离

prompt blind spot 已经影响评估器B的判断方式。

WK v1.2 只能学习 deterministic feature predicate。

不能把 prompt 文案直接搬进 WK。

错误示例：

```text
if "B thinks WK is harsh":
    upgrade to green
```

正确方向：

```text
if result == W
and opponent_quality in {top6, european_elite}
and xg_delta > 0.75
and shots_on_target_delta >= 2:
    candidate can propose upgrading a specific WK dimension/model
```

### 4.2 评估器B是 pseudo-gold，不是 human truth

三层标签继续保持：

```text
weak_labels      = deterministic WK 输出
evaluation       = 评估器B strict v2 pseudo-gold
human_override   = 明确人工修正
```

WK v1.2 候选可以参考评估器B。

但候选必须保留：

- examples
- counterexamples
- regression list
- replay impact
- missing feature behavior

### 4.3 只在 clean subset 上做主判断

b-003 的 8 条 quarantine 不能进入规则蒸馏主统计。

主判断使用：

```text
data/self_iteration/runs/b-003/adjudication_report.json
rows where status not in {missing_evaluator_b, invalid_evaluator_b}
```

当前应为 94 条。

## 8. Phase 2：蒸馏 WK v1.2 规则候选

新增命令：

```bash
python scripts/self_iterate.py distill-wk-rules \
  --kb data/knowledge.json \
  --baseline-adjudication data/self_iteration/runs/b-001/adjudication_report.json \
  --current-adjudication data/self_iteration/runs/b-003/adjudication_report.json \
  --comparison data/self_iteration/runs/b-003/comparison_report.json \
  --output data/self_iteration/runs/wk-v1.2/wk_rule_candidates.json
```

### 8.1 候选来源

候选只从 b-003 clean subset 中产生。

允许的 row status：

```text
wk_too_harsh
wk_too_generous
dimension_level_disagreement
model_level_disagreement
```

主目标优先级：

1. `wk_too_harsh`
2. `dimension_level_disagreement`
3. `model_level_disagreement`
4. `wk_too_generous`

解释：

- 当前主要剩余问题是 WK 仍偏严。
- `wk_too_generous` 必须更保守处理，避免重新放大 1531572 这类失败场景。

### 8.2 Candidate feature view

每个 match 需要派生一个稳定 feature view。

字段包括：

```json
{
  "match_id": "1208154",
  "result": "W",
  "opponent_quality": "top6",
  "venue": "home",
  "competition_stage": "league_early",
  "score_margin": 2,
  "clean_sheet": true,
  "dominant_control": true,
  "dominant_chance_quality": true,
  "low_defensive_risk": true,
  "narrow_win": false,
  "loss_despite_dominance": false,
  "xg_present": true,
  "missing_features": ["pressing", "transition"]
}
```

派生规则：

- `clean_sheet`: `goals_conceded == 0`
- `dominant_control`: 至少两个条件为真：
  - `shot_delta >= 5`
  - `possession_delta >= 8`
  - `corner_delta >= 4`
- `dominant_chance_quality`: `xg_delta >= 0.75`，缺 xG 时为 `false`
- `low_defensive_risk`: 至少两个条件为真：
  - `goals_conceded == 0`
  - `opponent_shots_on_target <= 3`
  - `xg_against <= 1.0`
- `narrow_win`: `result == W and score_margin == 1`
- `loss_despite_dominance`: `result == L and (dominant_control or dominant_chance_quality)`

缺失字段语义：

- 缺失字段不能让派生 boolean 变成 `true`。
- 每条 candidate 必须保留 `missing_features`。
- 不允许把“缺失数据”本身当成战术规律，除非 target 明确是 `missing_data_behavior`。

### 8.3 Candidate schema

```json
{
  "candidate_schema_version": "wk_rule_candidate_v1",
  "id": "wk_v1_2_top6_win_dominant_chance_quality_upgrade_satisfaction",
  "source_runs": ["b-001", "b-003"],
  "primary_run": "b-003",
  "target": "dimension_signals.satisfaction",
  "predicate": {
    "result": "W",
    "opponent_quality": ["top6", "european_elite"],
    "dominant_chance_quality": true,
    "low_defensive_risk": true
  },
  "current_wk_signal": "🟡",
  "target_signal": "🟢",
  "direction": "upgrade",
  "support": 6,
  "precision_vs_b": 0.83,
  "false_positive_count": 1,
  "examples": ["1208154", "1451107"],
  "counterexamples": ["1378993"],
  "regression_must_not_change": ["1531572"],
  "risk": "medium",
  "rationale": "评估器B在强队胜利且机会质量明显占优时稳定给更高满意度；WK v1.1 对这类胜利仍偏保守。",
  "implementation_hint": "Add a guarded satisfaction upgrade after base satisfaction but before overall voting."
}
```

### 8.4 晋级门槛

WK v1.2 candidate 必须满足：

- `support >= 5`
- `precision_vs_b >= 0.80`
- `false_positive_count <= 1`
- examples 覆盖至少 2 个不同对手
- candidate 有 deterministic feature predicate
- candidate 包含 regression list
- candidate 不与已有 known blind spot 冲突
- 对 `wk_too_generous` 方向的候选，必须额外满足：
  - `support >= 7`
  - `precision_vs_b >= 0.90`
  - `false_positive_count == 0`

如果候选依赖 b-001 与 b-003 两轮一致，记录：

```json
{
  "second_pass_support": true,
  "stable_across_runs": true
}
```

如果只在 b-003 中成立，记录：

```json
{
  "second_pass_support": false,
  "stable_across_runs": false
}
```

这类候选可以输出，但默认 `risk` 至少为 `medium`，且不能直接进入 implementation spec。

## 9. Phase 3：WK candidate dry-run replay

新增命令：

```bash
python scripts/self_iterate.py replay-wk-candidates \
  --kb data/knowledge.json \
  --adjudication data/self_iteration/runs/b-003/adjudication_report.json \
  --candidates data/self_iteration/runs/wk-v1.2/wk_rule_candidates.json \
```

行为：

- 不写 `data/knowledge.json`。
- 不修改 stored `weak_labels`。
- 在内存中模拟 WK v1.2 candidate 的影响。
- 只对 b-003 clean subset 计算主指标。
- 输出 per-match diff。

输出结构：

```json

    "wk_too_harsh": 0,
```

## 10. Phase 4：Regression manifest

生成：

```text
data/self_iteration/runs/wk-v1.2/regression_manifest.json
```

最低包含：

```json
{
  "must_not_become_green_overall": [
    {
      "match_id": "1531572",
      "reason": "dominant stats + loss to lower opposition must not become green"
    }
  ],
  "loss_guard": {
    "rule": "loss to lower/mid_table opposition cannot become overall green because of dominance metrics"
  },
  "win_regression_set": [],
  "quarantined_excluded": [
    "1208103",
    "1208118",
    "1208215",
    "1208233",
    "1208326",
    "1379169",
    "1379251",
    "1518728"
  ]
}
```

`win_regression_set` 应从 b-003 clean subset 中选择：

- top6 win
- european_elite win
- mid_table away win
- lower home win

每类最多 2 条。

作用：

- 防止 WK v1.2 为了解决偏严问题，又把失败场景放绿。
- 防止新规则把已经合理的胜利场景误降级。

不得直接修改 `weak_labeler.py`。

## 12. Dry-run 达标标准

WK v1.2 candidate replay 必须满足：

- clean subset `compared >= 90`
- `overall_agreement_rate` 不下降
- `dimension_agreement_rate` 不下降
- `model_agreement_rate` 不下降
- `wk_too_harsh` 相对下降至少 20%
- `wk_too_generous` 绝对增加不超过 2
- `1531572` 不得变成 overall 🟢
- lower/mid_table loss 不得因为 dominance metrics 变成 overall 🟢
- regression manifest 中所有 `must_not_change` 通过

未达标时：

- 不生成 WK implementation spec。
- 保留 replay artifact。
- 在 `rejected_candidates.json` 中写清失败原因。

## 13. 测试要求

### 13.1 compare-runs

补充测试：

- 分母相同时，直接输出 `effective`。
- 分母不同时，全量 `effective=false`。
- 分母不同时，输出 `clean_subset`。
- `clean_subset` 使用共同 compared match_ids。
- `clean_subset.effective=true` 时不改变全量 `effective=false`。

### 13.2 distill-wk-rules

新增测试：

- 只读取 b-003 clean subset。
- missing/quarantined rows 不进入 support。
- `support < 5` 的候选进入 `rejected_candidates`。
- `precision_vs_b < 0.80` 的候选进入 `rejected_candidates`。
- 缺失 xG 时 `dominant_chance_quality=false`。
- `wk_too_generous` 候选使用更高门槛。
- candidate schema 包含 regression list。

### 13.3 replay-wk-candidates

新增测试：

- replay 不修改 `data/knowledge.json`。
- candidate 只在 predicate 命中的 match 上生效。
- 1531572 regression 必须通过。
- lower/mid_table loss guard 必须通过。
- 输出 before/after metrics。

### 13.4 propose-wk-patch-spec

新增测试：

- replay 未达标时拒绝生成 implementation spec。
- replay 达标时生成 spec。
- 生成 spec 包含 candidate IDs、regression list、version bump 要求。

## 14. 验收标准

实现完成必须满足：

- 全量测试通过。
- `data/self_iteration/runs/wk-v1.2/validation_summary.json` 存在。
- `wk_rule_candidates.json` 存在，且候选结构符合 schema。
- `rejected_candidates.json` 存在，且保留未过门槛候选。
- `wk_candidate_replay.json` 存在，且 `dry_run=true`。
- `regression_manifest.json` 存在。
- 不修改 `data/knowledge.json` 中的 `weak_labels`。
- 不修改 `src/labels/weak_labeler.py`。
- 不新增 provider-specific runner 脚本。
- 若 replay 未达标，不生成 WK implementation spec。
- 若 replay 达标，生成单独的 implementation spec 等待人工审批。

## 15. Done Definition

本阶段完成时，应该能回答三个问题：

1. b-003 prompt blind spots 的有效性是否已经固化？
2. 哪些剩余 WK/B 分歧值得进入 WK v1.2 候选？
3. 这些候选在 dry-run 中是否提升 WK，而没有破坏关键 regression？

只有这三个问题都有 artifact 支撑，才能进入 WK v1.2 实现阶段。
